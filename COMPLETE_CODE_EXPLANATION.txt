PART 2: BACKEND - INFRASTRUCTURE & CONFIG

--- backend/server.js ---
This is the ENTRY POINT. When you run "node server.js", this file executes.

What it does step by step:
  1. Creates an HTTP server from the Express app (src/app.js)
  2. Calls connectDB() to connect to MongoDB
  3. Calls connectRedis() to connect to Redis (optional, continues if fails)
  4. Calls initSocket(server, redisClient) to set up Socket.IO on the HTTP server
     - Passes the Redis client so Socket.IO can use Redis as a pub/sub adapter
     - This allows multiple server instances to share socket events
  5. Stores the Socket.IO instance on the Express app: app.set('io', io)
     - This lets controllers access io to emit events (e.g., new-match)
  6. Calls initFirebase() - initializes Firebase Admin SDK (for push notifications)
  7. Calls initCronJobs() - starts scheduled background tasks
  8. server.listen(port) - starts accepting HTTP connections

  Shutdown handler:
  - Listens for SIGTERM (Docker stop) and SIGINT (Ctrl+C)
  - Closes HTTP server (stops accepting new connections)
  - Closes MongoDB connection
  - Closes Redis connection
  - Force exits after 10 seconds if graceful shutdown hangs

  Error handlers:
  - uncaughtException: Logs the error and triggers shutdown
  - unhandledRejection: Logs the error (does not crash, just warns)


--- backend/src/app.js ---
This creates and configures the Express application.

Middleware applied IN ORDER (order matters!):
  1. Stripe webhook route (raw body) - BEFORE json parser
     - Stripe needs the raw request body to verify webhook signatures
     - If we parse JSON first, the signature verification fails

  2. helmet() - Sets security HTTP headers:
     - X-Content-Type-Options: nosniff
     - X-Frame-Options: DENY
     - Strict-Transport-Security (HSTS)
     - Content-Security-Policy
     - And many more security headers

  3. cors() - Cross-Origin Resource Sharing:
     - In development: allows all origins (for easy testing)
     - In production: only allows origins listed in CORS_ORIGINS env var
     - Credentials: true (allows cookies/auth headers)
     - Preflight cache: 24 hours (reduces OPTIONS requests)

  4. compression() - Gzip response compression (smaller payloads)

  5. express.json({ limit: '10mb' }) - Parses JSON request bodies
     - Limit prevents someone sending a 1GB JSON body to crash the server

  6. express.urlencoded({ extended: true }) - Parses URL-encoded form data

  7. sanitize middleware - Strips MongoDB operators ($gt, $ne, etc.) from input
     - Prevents NoSQL injection attacks
     - Example: someone sends { "email": { "$gt": "" } } to match all users
     - Sanitizer removes the $ key, blocking the attack

  8. morgan - HTTP request logging (dev: colored, prod: structured to pino)

  9. Request timeout (30 seconds) - If a request takes longer, returns 408

Routes:
  - GET /health - Deep health check (checks MongoDB + Redis status)
  - GET /ready - Readiness probe for Kubernetes (just checks MongoDB)
  - /api/v1/* - All API routes (see routes/index.js)
  - 404 handler - Catches any unmatched route
  - Sentry error handler - Captures errors to Sentry (if configured)
  - Global error handler - Formats and returns error responses


--- backend/src/config/index.js ---
Central configuration. Reads .env file and exports a frozen (immutable) object.

Required variables (server won't start without these):
  - MONGO_URI: MongoDB connection string
  - JWT_ACCESS_SECRET: Secret key for signing access tokens
  - JWT_REFRESH_SECRET: Secret key for signing refresh tokens

All other variables have sensible defaults. If SMTP is not configured,
OTP codes are logged to console in development mode.

The config object is frozen with Object.freeze() so no code can accidentally
modify configuration at runtime.


--- backend/src/config/db.js ---
MongoDB connection using Mongoose.

Connection pool settings:
  - Production: maxPoolSize=50, minPoolSize=10
  - Development: maxPoolSize=10, minPoolSize=2

A "connection pool" means Mongoose keeps multiple connections open to MongoDB.
When your code does a database query, it grabs an available connection from the
pool instead of opening a new one. This is faster because opening connections
is expensive. maxPoolSize=50 means up to 50 simultaneous queries can run.

If MongoDB disconnects (network issue, server restart), Mongoose automatically
reconnects. The old code had process.exit(1) which would crash the server.
Now it just logs a warning and waits for Mongoose to reconnect.


--- backend/src/config/redis.js ---
Redis connection using ioredis library.

Redis is an in-memory data store. We use it for:
  1. Caching feed results (so we don't re-run the expensive aggregation)
  2. Socket.IO pub/sub adapter (so multiple server instances share events)

The connection uses lazy connect (doesn't connect until first use) and has
a retry strategy: retries 5 times with increasing delays, then stops.

If Redis is not available, the app continues to work. Feed results are just
fetched from MongoDB every time, and Socket.IO runs in single-instance mode.


--- backend/src/config/cloudinary.js ---
Cloudinary is a cloud image hosting service. When users upload photos:
  1. Photo is uploaded to Cloudinary (not stored on our server)
  2. Cloudinary returns a URL and a publicId
  3. We store the URL (for displaying) and publicId (for deleting later)

Cloudinary also handles image transformations (resize, quality optimization)
automatically during upload.


--- backend/src/config/firebase.js ---
Firebase Admin SDK for sending push notifications.

Requires a service account JSON file (downloaded from Firebase Console).
If the file path isn't set, push notifications are silently disabled.
This is intentional - you can develop without Firebase set up.


--- backend/src/config/sentry.js ---
Sentry is an error tracking service. When an unhandled error occurs:
  1. Sentry captures the error, stack trace, and request context
  2. Sends it to sentry.io where you can see it in a dashboard
  3. Can alert you via email/Slack when new errors appear

tracesSampleRate: 0.1 means 10% of requests are traced for performance
monitoring. In development, 100% are traced.

Sensitive data (authorization headers, cookies) is scrubbed before sending.


--- backend/src/config/stripe.js ---
Stripe is the payment processor for paid boosts.
If STRIPE_SECRET_KEY is not set, the stripe client is null, and
any attempt to purchase a boost will return "Payments not configured".

PART 3: BACKEND - DATABASE MODELS (MongoDB/Mongoose)
================================================================================

MongoDB stores data in "collections" (like tables in SQL). Each collection
has a "schema" that defines the structure. Mongoose is the Node.js library
that provides schemas, validation, and query building for MongoDB.


--- backend/src/models/User.js ---
The central model. Every person who uses the app has a User document.

Fields:
  email         String, unique, lowercase, trimmed
                The user's email address. Used for OTP login.
                "unique: true" creates a database index automatically.

  googleId      String, sparse index
                Google OAuth user ID. "sparse" means the index only includes
                documents where this field exists (not all users have Google).

  name          String, max 50 characters
                Display name shown on profile cards.

  age           Number, minimum 18, maximum 100
                Age validation ensures no minors can register.

  gender        String, enum ['male', 'female']
                This determines the entire app experience:
                - 'female' -> gets swipe UI (browse boys)
                - 'male'   -> gets queue UI (see who liked them)

  bio           String, max 300 characters
                Short description shown on profile cards.

  interests     Array of Strings
                Tags like "Music", "Travel", "Fitness" etc.
                Shown on profile cards.

  photos        Array of { url: String, publicId: String }
                - url: Cloudinary URL for displaying the image
                - publicId: Cloudinary ID for deleting the image
                Minimum 2 photos required for profile completion.

  location      GeoJSON Point object
                - type: always 'Point' (GeoJSON standard)
                - coordinates: [longitude, latitude] (NOTE: lon first, NOT lat!)
                - city: String (for display)
                - state: String (for display)
                The coordinates enable geo-spatial queries (find users near me).

  preferences   Object
                - ageMin: minimum age of matches to show (default 18)
                - ageMax: maximum age of matches to show (default 50)
                - maxDistance: maximum distance in km (default 50)

  daysWithoutMatch   Number, default 0
                Incremented daily by the cron job for all active boys.
                Reset to 0 when a match occurs.
                Used to calculate auto-boost tier.

  boostLevel    String, enum ['none', 'bronze', 'silver', 'gold'], default 'none'
                Current PAID boost level. Different from auto-boost.

  boostExpiry   Date
                When the paid boost expires. After this date, boostLevel
                reverts to 'none' (done by the hourly cron job).

  fcmToken      String
                Firebase Cloud Messaging token. Each device has a unique token.
                Used to send push notifications to the user's phone.

  refreshToken  String
                SHA-256 hash of the user's refresh token.
                NOT stored in plain text (security measure).
                Used to issue new access tokens when the old one expires.

  isProfileComplete   Boolean, default false
                Set to true when user has: name, age, gender, 2+ photos,
                bio, and 1+ interests. Users with incomplete profiles
                don't appear in the feed.

  isActive      Boolean, default true
                Can be set to false to hide user from feed without deleting.

  timestamps    true (Mongoose option)
                Automatically adds createdAt and updatedAt fields.

Indexes (for query performance):
  - location: '2dsphere' -> enables $geoNear queries (find nearby users)
  - { gender, isActive, isProfileComplete } -> for feed filtering
  - { boostLevel, boostExpiry } -> for expired boost cleanup
  - { daysWithoutMatch: -1 } -> for feed sorting (descending)


--- backend/src/models/Like.js ---
Represents a girl liking (or skipping) a boy.

Fields:
  fromUser    ObjectId -> User (the girl who liked)
  toUser      ObjectId -> User (the boy who was liked)
  status      String, enum:
              - 'pending'  -> girl liked, boy hasn't responded yet
              - 'accepted' -> boy accepted, match created
              - 'rejected' -> boy rejected, girl NOT notified
              - 'skipped'  -> girl swiped left (skip, not a like)

Indexes:
  - { fromUser, toUser } UNIQUE -> prevents duplicate likes
    If a girl tries to like the same boy twice, MongoDB throws error 11000.
  - { toUser, status } -> fast queue lookup for boys
    "Show me all pending likes where toUser = this boy's ID"


--- backend/src/models/Match.js ---
Created when a boy accepts a girl's like.

Fields:
  users     Array of 2 ObjectIds -> User
            Always contains exactly 2 users (the girl and the boy).

  timestamps -> createdAt = when the match happened

Index:
  - { users: 1 } -> fast lookup "find all matches containing this user ID"


--- backend/src/models/Message.js ---
Chat messages between matched users.

Fields:
  matchId   ObjectId -> Match
            Which match/conversation this message belongs to.

  sender    ObjectId -> User
            Who sent this message.

  text      String, max 2000 characters
            The message content.

  seen      Boolean, default false
            Read receipt. Set to true when the recipient views the message.

Index:
  - { matchId, createdAt } -> fast retrieval of messages in a conversation
    ordered by time.


--- backend/src/models/Otp.js ---
Temporary one-time passwords for email verification.

Fields:
  email     String (which email this OTP is for)
  code      String (the 6-digit code)
  expiresAt Date (when the code becomes invalid)

Index:
  - { expiresAt } with expireAfterSeconds: 0
    This is a MongoDB TTL (Time-To-Live) index. MongoDB automatically
    deletes documents after their expiresAt date passes.
    So expired OTPs are cleaned up without any code needed.
  - { email } -> fast lookup by email


--- backend/src/models/Block.js ---
When user A blocks user B.

Fields:
  blocker   ObjectId -> User (who did the blocking)
  blocked   ObjectId -> User (who was blocked)

Indexes:
  - { blocker, blocked } UNIQUE -> can't block the same person twice
  - { blocked, blocker } -> reverse lookup (who blocked me?)

When a block is created, the safety service also:
  - Deletes any existing match between the two users
  - Deletes any pending likes between the two users


--- backend/src/models/Report.js ---
When a user reports another user for bad behavior.

Fields:
  reporter  ObjectId -> User
  reported  ObjectId -> User
  reason    String, enum: 'harassment', 'fake', 'inappropriate', 'spam', 'other'
  details   String, max 500 chars (optional extra info)

Reports are stored for admin review. No automatic action is taken.


================================================================================
PART 4: BACKEND - MIDDLEWARE
================================================================================

Middleware are functions that run BEFORE your route handler. They sit in the
"middle" of the request pipeline. Each middleware can:
  - Modify the request (add data, parse body)
  - End the request (return error, unauthorized)
  - Pass to the next middleware (call next())


--- backend/src/middleware/auth.middleware.js ---
JWT authentication. Runs on all protected routes.

How it works:
  1. Reads the "Authorization" header from the request
  2. Expects format: "Bearer <token>"
  3. Extracts the token part after "Bearer "
  4. Calls verifyAccessToken(token) which does jwt.verify()
     - If token is valid: returns { id, gender } from the token payload
     - If token is expired: throws TokenExpiredError
     - If token is tampered: throws JsonWebTokenError
  5. Attaches decoded data to req.user = { id, gender }
  6. Calls next() to continue to the route handler

After this middleware runs, any route handler can access:
  - req.user.id -> the authenticated user's MongoDB _id
  - req.user.gender -> 'male' or 'female'


--- backend/src/middleware/gender.middleware.js ---
Gender-based access control. A "factory function" that returns middleware.

Usage in routes/index.js:
  router.use('/swipe', auth, requireGender('female'), swipeRoutes);
  router.use('/queue', auth, requireGender('male'), queueRoutes);

How it works:
  1. requireGender('female') returns a middleware function
  2. That middleware checks if req.user.gender === 'female'
  3. If yes: calls next() (proceed)
  4. If no: returns 403 Forbidden error

This enforces the core app rule:
  - Only girls can browse/swipe (swipe routes)
  - Only boys can see queue/accept/reject (queue routes)
  - Only boys can purchase boosts (boost routes)


--- backend/src/middleware/validate.middleware.js ---
Joi schema validation for request body and query parameters.

How it works:
  1. Takes a Joi schema as argument
  2. Returns a middleware function
  3. That middleware validates req.body (or req.query) against the schema
  4. Options:
     - abortEarly: false -> collect ALL errors, not just the first
     - stripUnknown: true -> remove fields not in the schema
  5. If validation fails: returns 400 with error messages
  6. If passes: replaces req.body with the validated/cleaned value

Example:
  Schema: { email: Joi.string().email().required() }
  Input:  { email: "test@test.com", hack: "drop table" }
  Result: { email: "test@test.com" }  <- "hack" field stripped


--- backend/src/middleware/sanitize.middleware.js ---
NoSQL injection prevention.

MongoDB uses $ prefix for operators: $gt (greater than), $ne (not equal),
$where (arbitrary JavaScript), etc.

Attack example without sanitization:
  POST /auth/verify-otp
  Body: { "email": {"$gt": ""}, "code": {"$gt": ""} }

  This would match ANY email and ANY code in the database!
  The attacker could verify without knowing the real OTP.

How the sanitizer works:
  1. Recursively walks through req.body, req.query, req.params
  2. If any object key starts with '$', it removes that key
  3. Handles nested objects and arrays

After sanitization:
  { "email": {"$gt": ""} } becomes { "email": {} } -> harmless


--- backend/src/middleware/rateLimiter.middleware.js ---
Rate limiting prevents abuse by limiting how many requests a client can make.

Rate limiters defined:

  globalLimiter:
    - 1000 requests per 15 minutes per IP
    - Applied to ALL API routes
    - Prevents general API abuse

  authLimiter:
    - 10 requests per 15 minutes per IP
    - Applied to signup, verify-otp, google auth
    - Prevents brute force login attempts

  otpSendLimiter:
    - 3 requests per 15 minutes per EMAIL
    - Applied to POST /auth/signup (sending OTP)
    - Prevents spamming someone's email with OTP codes
    - Key: "otp-send:user@email.com"

  otpVerifyLimiter:
    - 5 requests per 15 minutes per EMAIL
    - Applied to POST /auth/verify-otp
    - Prevents brute-forcing OTP codes
    - 6-digit OTP = 1,000,000 possibilities
    - At 5 attempts per 15 minutes, brute force is impossible
    - Key: "otp:user@email.com"

  likeLimiter:
    - 100 requests per hour per USER
    - Applied to POST /swipe/like
    - Prevents a girl from mass-liking all boys

  messageLimiter:
    - 200 requests per hour per USER
    - Applied to POST /messages
    - Prevents message flooding/spam


--- backend/src/middleware/upload.middleware.js ---
Multer configuration for file uploads.

Settings:
  - storage: memoryStorage() -> files stored in RAM buffer (not disk)
    We use memory because we immediately upload to Cloudinary
  - fileFilter: only accepts image/jpeg, image/png, image/webp
  - limits:
    - fileSize: 5MB per file
    - files: 6 maximum (matches max photos per user)

Note: This only checks the MIME type header. The upload service also
validates magic bytes (actual file content) for extra security.


--- backend/src/middleware/error.middleware.js ---
Global error handler. This is the LAST middleware in the chain.
Express recognizes it as an error handler because it has 4 parameters
(err, req, res, next).

It catches all errors thrown by route handlers and transforms them
into consistent JSON responses.

Error types handled:
  - Mongoose ValidationError -> 400 (bad input)
  - Mongoose duplicate key (11000) -> 409 (conflict, already exists)
  - Mongoose CastError -> 400 (invalid ObjectId format)
  - JsonWebTokenError -> 401 (bad token)
  - TokenExpiredError -> 401 (expired token)
  - AppError (our custom) -> whatever statusCode was set
  - Everything else -> 500 (internal server error)

In production: returns { error: { code, message } }
In development: also includes the stack trace for debugging

--- backend/src/services/auth.service.js ---

sendOtp(email):
  1. Generate cryptographically random 6-digit code using crypto.randomInt()
  2. Calculate expiry time: now + OTP_EXPIRY_MINUTES (default 5)
  3. Delete any existing OTPs for this email (user requested new one)
  4. Create new OTP document in database
  5. Send email with the code (or log to console in dev mode)

verifyOtp(email, code):
  1. Find OTP document matching email AND code
  2. If not found -> "Invalid or expired OTP"
  3. If found but expired -> "OTP expired"
  4. Delete all OTPs for this email (clean up)
  5. Find user by email, or create new user if not found
  6. Call issueTokens() to generate access + refresh tokens
  7. Return { accessToken, refreshToken, user, isNewUser }

googleLogin(idToken):
  1. Verify the Google ID token using Google's library
     (This contacts Google's servers to validate the token is real)
  2. Extract: googleId, email, name from the token
  3. Find user by googleId OR email
  4. If not found: create new user
  5. If found but no googleId: link Google account
  6. Call issueTokens()

issueTokens(user, isNewUser):
  1. Generate access token: JWT containing { id, gender }, expires in 15 min
  2. Generate refresh token: JWT containing { id }, expires in 7 days
  3. Hash the refresh token with SHA-256
  4. Store the HASH in the database (not the plain token)
  5. Return both tokens and user data

  WHY hash the refresh token?
  If an attacker gains read access to your database (SQL injection, backup leak),
  they see hashed tokens which are useless. They can't create valid JWTs from
  a hash. Without hashing, they'd have valid refresh tokens for every user.

refreshTokens(token):
  1. Verify the JWT is valid (not expired, not tampered)
  2. Find user by ID from the token
  3. Hash the incoming token with SHA-256
  4. Compare with stored hash
  5. If mismatch -> possible token reuse attack!
     - Clear the user's stored token (invalidate ALL sessions)
     - Return error
  6. If match -> issue new token pair, store new hash

  TOKEN ROTATION: Each refresh generates a NEW refresh token.
  The old one is invalidated. If an attacker steals a refresh token
  and the real user also uses it, one of them will get a mismatch,
  triggering the security measure of invalidating everything.

--- backend/src/services/match.service.js ---

getMatches(userId, page, limit):
  Uses a SINGLE MongoDB aggregation pipeline instead of N+1 queries.

  Old approach (N+1 problem):
    1. Find all matches for user (1 query)
    2. For EACH match, find last message (N queries)
    3. For EACH match, count unread messages (N queries)
    = 1 + 2N queries. With 100 matches = 201 queries!

  New approach (single aggregation):
    1. $match: find matches containing this user
    2. $lookup messages: get last message for each match (sub-pipeline)
    3. $lookup messages: count unread for each match (sub-pipeline)
    4. $lookup users: populate user details
    5. $addFields: reshape data
    6. $sort by most recent activity
    7. $skip + $limit for pagination
    = 1 query total, no matter how many matches!

--- backend/src/services/upload.service.js ---

validateImageContent(buffer):
  Checks the first 12 bytes of the file against known "magic bytes":
  - JPEG: starts with FF D8 FF
  - PNG: starts with 89 50 4E 47
  - WebP: starts with RIFF....WEBP

  WHY? The upload middleware checks the MIME type header, but that's
  set by the client and can be faked. An attacker could upload a
  malicious script file with a fake "image/jpeg" MIME type. Checking
  the actual bytes ensures it's really an image.

uploadImage(buffer, userId):
  1. Validate magic bytes
  2. Upload to Cloudinary as a stream
  3. Cloudinary applies transformations:
     - Resize: max 800x1000 pixels
     - Quality: auto (Cloudinary optimizes)
     - Format: auto (serves WebP to supporting browsers)
  4. Return { url, publicId }

--- backend/src/utils/AppError.js ---
Custom error class extending built-in Error.

Properties:
  - message: human-readable error message
  - statusCode: HTTP status code (400, 401, 403, 404, 500, etc.)
  - isOperational: true (distinguishes from programming bugs)

Usage: throw new AppError('User not found', 404);

The error middleware checks isOperational to decide whether to log the error.
Operational errors (bad input, not found) are expected. Non-operational errors
(null pointer, undefined function) are bugs that should be logged in detail.


--- backend/src/utils/catchAsync.js ---
Wraps async route handlers so errors are automatically caught.

Without catchAsync:
  router.get('/profile', async (req, res, next) => {
    try {
      const profile = await getProfile(req.user.id);
      res.json(profile);
    } catch (err) {
      next(err);  // Must remember this or errors crash the server!
    }
  });

With catchAsync:
  router.get('/profile', catchAsync(async (req, res) => {
    const profile = await getProfile(req.user.id);
    res.json(profile);
  }));

catchAsync wraps the function and catches any rejected promise,
passing the error to next() automatically.


--- backend/src/utils/logger.js ---
Structured JSON logging using Pino.

Why Pino over console.log?
  - JSON format: parseable by log aggregators (Datadog, ELK, CloudWatch)
  - Levels: error, warn, info, debug (can filter by level)
  - Redaction: automatically strips sensitive data from logs
  - Performance: Pino is the fastest Node.js logger (async I/O)

Configuration:
  - Production: JSON output, 'info' level (no debug messages)
  - Development: pretty-printed colored output, 'debug' level
  - Redacted paths: authorization header, password, refreshToken

Example output (production):
  {"level":30,"time":1234567890,"msg":"Server started","port":5000,"env":"production"}


--- backend/src/utils/token.js ---

signAccessToken(userId, gender):
  Creates a JWT with payload { id: userId, gender: gender }
  Signed with JWT_ACCESS_SECRET
  Expires in JWT_ACCESS_EXPIRY (default 15 minutes)

  Why include gender? So the auth middleware can attach it to req.user
  without an extra database query on every request.

signRefreshToken(userId):
  Creates a JWT with payload { id: userId }
  Signed with JWT_REFRESH_SECRET (different key from access!)
  Expires in JWT_REFRESH_EXPIRY (default 7 days)

  Why a different secret? If the access secret is compromised,
  the refresh token is still safe (and vice versa).

verifyAccessToken(token):
  Calls jwt.verify() with the access secret.
  Returns the decoded payload or throws an error.

verifyRefreshToken(token):
  Same but with the refresh secret.

hashRefreshToken(token):
  SHA-256 hash of the token string.
  Used for secure storage in the database.
  SHA-256 is sufficient because refresh tokens are already high-entropy
  (they're JWTs with random data). We don't need bcrypt's slow hashing
  which is designed for low-entropy passwords.


--- backend/src/utils/otp.js ---

generateOtp():

sendOtpEmail(email, code):
  1. In development with no SMTP configured: logs the OTP to console

  2. In production: sends an HTML email via nodemailer:

PART 8: BACKEND - SOCKET.IO (Real-time Chat)
================================================================================

Socket.IO enables bidirectional real-time communication. Unlike HTTP where the
client always initiates, with WebSockets the server can push data to clients.

--- backend/src/socket/index.js ---

initSocket(httpServer, redisClient):
  1. Create Socket.IO server attached to the HTTP server
  2. Configure CORS (same rules as Express)
  3. Set maxHttpBufferSize (64KB default) - limits message payload size
  4. If Redis is available: create Redis adapter

     WHY Redis adapter?
     Without it, if you have 2 server instances:
     - User A connects to Server 1
     - User B connects to Server 2
     - User A sends a message to User B
     - Server 1 has no idea User B exists (they're on Server 2!)

     With Redis adapter:
     - Server 1 publishes the event to Redis
     - Server 2 subscribes to Redis and receives the event
     - Server 2 sends it to User B
     - Now both servers share real-time events!

  5. JWT authentication middleware for sockets:
     - Client sends token in socket.handshake.auth.token
     - Server verifies the JWT
     - Attaches user data to socket.user
     - If invalid: connection rejected

  6. On connection:
     - Join a "personal room" with the user's ID
       This means io.to(userId).emit(...) works even without
       tracking socket IDs manually
     - Set up chat handlers
     - On disconnect: log it

--- backend/src/socket/chat.handler.js ---

Events the CLIENT sends:

  'join-room' (matchId):
    Client wants to enter a chat room.
    Server validates: match exists AND client is a participant.
    If valid: socket.join(matchId)
    Now client receives all events emitted to that room.

  'leave-room' (matchId):
    Client leaves the chat room (navigating away from chat screen).

  'send-message' ({ matchId, text }):
    Client sends a chat message.
    Server validates:
    - matchId is a non-empty string
    - text is a non-empty string
    - text is under 2000 characters
    Then: saves to database, broadcasts to all in the room.

    io.to(matchId).emit('new-message', message)
    This sends to ALL sockets in the room, INCLUDING the sender.
    The sender uses this as confirmation that the message was saved.

  'typing-start' (matchId):
    Client started typing. Server broadcasts to others in the room.
    (NOT to the sender - uses socket.to() instead of io.to())

  'typing-stop' (matchId):
    Client stopped typing.

  'mark-seen' ({ matchId }):
    Client has read the messages. Server:
    1. Updates all unread messages in DB (seen = true)
    2. Broadcasts 'messages-seen' to others in the room

Events the SERVER sends:

  'new-message': A message was sent in a room you're in
  'user-typing': Someone started typing in your chat
  'user-stopped-typing': Someone stopped typing
  'messages-seen': Someone read your messages (blue checkmarks)
  'new-like': (sent to boy's personal room) Someone liked your profile
  'new-match': (sent to both users' rooms) A match was created
  'error': Something went wrong with your last action

--- backend/src/jobs/dailyBoost.job.js ---

Uses node-cron to schedule recurring tasks.
Cron syntax: "minute hour day month weekday"

Job 1: Daily boost increment
  Schedule: "0 0 * * *" = every day at midnight UTC

  What it does:
  For all male users who are active and have complete profiles,
  increment daysWithoutMatch by 1.

  This is BATCHED: processes 5000 users at a time to avoid
  overwhelming MongoDB with a single massive update on millions of users.

  The batch uses cursor-based pagination (lastId) to iterate through
  all qualifying users:
    1. Find 5000 users where _id > lastId
    2. Update those 5000 users: daysWithoutMatch += 1
    3. Remember the last _id
    4. Repeat until no more users

  Why this matters:
  daysWithoutMatch determines the auto-boost tier:
    0-6 days:   none   (normal visibility)
    7-14 days:  bronze (higher in feed)
    15-24 days: silver (even higher)
    25+ days:   gold   (top of feed)

  This ensures boys who haven't been getting matches get more visible
  over time, giving everyone a chance.

Job 2: Expired boost cleanup
  Schedule: "0 * * * *" = every hour at minute 0

  What it does:
  Find all users where:
  - boostLevel is NOT 'none' (they have a paid boost)
  - boostExpiry is in the past (the boost has expired)
  Set boostLevel back to 'none' and clear boostExpiry.

--- reverse_match/lib/core/network/dio_client.dart ---
HTTP client setup using Dio library.

Configuration:
  - Base URL: http://localhost:5000/api/v1
  - Connect timeout: 15 seconds
  - Receive timeout: 15 seconds
  - Content-Type: application/json

INTERCEPTOR (the key part):
  onRequest:
    Before every HTTP request, reads the access token from secure storage
    and adds it as "Authorization: Bearer <token>" header.

  onError (401 handling):
    If ANY request returns 401 (unauthorized):
    1. Read the refresh token from storage
    2. Call /auth/refresh-token with a SEPARATE Dio instance
       (separate to avoid infinite loop - the main dio would
        add the expired token again!)
    3. If refresh succeeds:
       - Save new tokens to storage
       - Retry the original failed request with the new token
    4. If refresh fails:
       - Clear all tokens (user must login again)
       - Return UnauthorizedException

  This means the user NEVER sees token expiry. Access tokens expire
  every 15 minutes but are silently refreshed in the background.


--- reverse_match/lib/core/network/api_result.dart ---
A sealed class for representing API call results:

  sealed class ApiResult<T>
    |-- Success<T>(T data)
    |-- Failure<T>(AppException exception)

Usage with Dart 3 pattern matching:
  final result = await repo.getProfile();
  switch (result) {
    case Success(:final data):
      // use data
    case Failure(:final exception):
      // show exception.message
  }

This forces EVERY call site to handle both success and failure.
You literally can't forget to handle errors because the compiler
will warn about non-exhaustive switch.


--- reverse_match/lib/core/network/api_exceptions.dart ---
Exception hierarchy:
  AppException (base) - message + optional statusCode
    |-- NetworkException (no internet)
    |-- ServerException (server returned an error)
    |-- UnauthorizedException (401, session expired)
    |-- TimeoutException (request took too long)

auth_repository.dart:
  sendOtp(email):
    POST /auth/signup with { email }
    Returns Success("OTP sent") or Failure with error message

  verifyOtp(email, code):
    POST /auth/verify-otp with { email, code }
    On success: saves tokens + userId + gender to secure storage
    Returns Success(AuthResult) or Failure

    POST /auth/google with { idToken }
    Same save behavior as verifyOtp

  logout():
    POST /auth/logout (best effort, ignores errors)
    Clears all secure storage

auth_state.dart:
  Sealed class with 6 states:
    AuthInitial       -> app just started, haven't checked yet
    AuthLoading       -> checking auth or sending OTP
    AuthOtpSent(email) -> OTP was sent, show verify screen
    AuthAuthenticated(user, isNewUser) -> logged in
    AuthUnauthenticated -> not logged in
    AuthError(message) -> something went wrong

auth_provider.dart (AuthNotifier):
  The central state manager for authentication.

  checkAuthStatus():
    Called on app startup from splash screen.
    Reads stored tokens. If present, connects socket and sets authenticated.

  sendOtp(email):
    Sets loading, calls repo.sendOtp, transitions to OtpSent or Error.

  verifyOtp(email, code):
    Sets loading, calls repo.verifyOtp.
    On success: connects socket, sets Authenticated with user data.
    The router automatically redirects based on profile completeness.

  signInWithGoogle():
    Uses google_sign_in package to show Google's sign-in UI.
    Gets idToken from Google, sends to our backend for verification.

  logout():
    Disconnects socket, calls repo.logout (clears storage).
    Sets Unauthenticated. Router redirects to /welcome.

  clearError():
    Transitions from Error back to Unauthenticated.

--- backend/ecosystem.config.js ---
PM2 configuration for production deployment WITHOUT Docker.

PM2 is a Node.js process manager that provides:
  - Cluster mode: runs multiple instances (one per CPU core)
  - Auto-restart on crash
  - Log management
  - Memory limits (restart if exceeds 500MB)

Configuration:
  instances: 'max' -> use ALL CPU cores
  exec_mode: 'cluster' -> fork one process per core

  With a 4-core machine: 4 Node.js processes running simultaneously,
  sharing the same port. The OS load balancer distributes requests.

  This is why we need Redis for Socket.IO - each process is independent
  and doesn't know about connections on other processes.

PART 18: HOW THE BOOST SYSTEM WORKS
================================================================================

The boost system is the app's core differentiator. It ensures every boy
eventually gets visibility, even if girls initially skip him.

TWO TYPES OF BOOST:

1. AUTO-BOOST (free, automatic):
   Based on daysWithoutMatch counter.

   Day 0-6:   No boost     (boostScore = 0)
   Day 7-14:  Bronze boost  (boostScore = 200)
   Day 15-24: Silver boost  (boostScore = 500)
   Day 25+:   Gold boost    (boostScore = 1000)

   The daily cron job increments daysWithoutMatch at midnight UTC
   for ALL active male users with complete profiles.

   When a match occurs (boy accepts a like), daysWithoutMatch
   resets to 0. The boy starts over at "no boost".

2. PAID BOOST (purchased via Stripe):
   Bronze: $4.99, 1 day, boostScore = 200
   Silver: $9.99, 3 days, boostScore = 500
   Gold:   $19.99, 7 days, boostScore = 1000

   Stored as boostLevel + boostExpiry on the User document.
   Hourly cron job clears expired boosts.

PRIORITY:
   Paid boost takes priority over auto-boost while active.
   If paid boost expires, auto-boost kicks back in.

   Example: Boy has daysWithoutMatch=20 (auto silver).
   He buys gold boost. For 7 days he has gold.
   After 7 days, paid boost expires, auto silver resumes.

HOW IT AFFECTS THE FEED:
   The feed aggregation pipeline calculates an "effectiveBoost" for each boy:

   If (boostLevel != 'none' AND boostExpiry > now):
     effectiveBoost = boostLevel (paid)
   Else:
     effectiveBoost = auto-boost based on daysWithoutMatch

   Then converts to numeric boostScore:
     gold=1000, silver=500, bronze=200, none=0

   Feed is sorted by: boostScore DESC, daysWithoutMatch DESC, _id DESC

   This means gold boys appear FIRST, then silver, then bronze, then normal.
   Within the same boost tier, boys with more days without match appear first.

   Effectively, the system creates a queue where the "most in need" boys
   get the most visibility.

PART 19: HOW THE MATCHING ALGORITHM WORKS
================================================================================

The feed algorithm (swipe.service.js getFeed) is the most complex piece
of the backend. Here's the full pipeline explained:

STEP 1: CACHE CHECK
  If this is the first page (no cursor), check Redis cache.
  Cache key: "feed:{userId}:{limit}"
  TTL: 2 minutes
  If cached: return immediately (skip entire pipeline)

STEP 2: LOAD GIRL'S PREFERENCES
  Find the authenticated girl user.
  Extract: ageMin, ageMax, maxDistance, location coordinates.

STEP 3: BUILD EXCLUSION LIST
  Run two queries in parallel:
  a) Like.find({ fromUser: girlId }).distinct('toUser')
     -> All boys she's already interacted with (liked, skipped)
  b) Block.find({ blocker: girlId OR blocked: girlId })
     -> All users she's blocked or who blocked her

  Combine into a Set of unique IDs to exclude.
  Also add the girl's own ID (can't show yourself).
  Convert to ObjectId array for MongoDB query.

STEP 4: GEO FILTER (if girl has location)
  Uses MongoDB's $geoNear aggregation stage:
  - near: girl's coordinates
  - maxDistance: girl's maxDistance preference * 1000 (km to meters)
  - spherical: true (Earth is round!)
  - query filter: excludeIds, gender=male, isActive, isProfileComplete,
                  age between ageMin and ageMax
  - Adds a "distance" field to each result (meters from the girl)

  If girl has NO location (coordinates are [0,0]):
  Uses regular $match with the same filters but no geo constraint.

STEP 5: CALCULATE EFFECTIVE BOOST
  For each remaining boy, determine their boost tier:

  $addFields: effectiveBoost =
    IF boostLevel != 'none' AND boostExpiry > now:
      boostLevel (paid boost is active)
    ELSE:
      IF daysWithoutMatch >= 25: 'gold'
      IF daysWithoutMatch >= 15: 'silver'
      IF daysWithoutMatch >= 7:  'bronze'
      ELSE: 'none'

STEP 6: CALCULATE BOOST SCORE (numeric)
  $addFields: boostScore =
    gold=1000, silver=500, bronze=200, none=0

STEP 7: CURSOR PAGINATION
  If a cursor (last seen _id) was provided:
  $match: _id < cursor
  This gets the "next page" after the cursor.

STEP 8: SORT
  $sort: { boostScore: -1, daysWithoutMatch: -1, _id: -1 }
  Primary: highest boost first
  Secondary: most days without match first
  Tertiary: newest profile first

STEP 9: LIMIT
  $limit: 20 (or whatever limit was requested)

STEP 10: PROJECT (select fields)
  Only return: name, age, bio, interests, photos, location(city/state),
               effectiveBoost, distance

STEP 11: CACHE AND RETURN
  If first page: cache the result for 2 minutes.
  Return: { profiles: [...], nextCursor: last_id or null }

  nextCursor is null when there are fewer results than the limit
  (meaning no more pages).

PART 20: SECURITY MEASURES EXPLAINED
================================================================================

1. JWT WITH REFRESH TOKEN ROTATION
   Access tokens expire in 15 minutes (short-lived, limits exposure).
   Refresh tokens are rotated: each use generates a new pair.
   Old refresh tokens become invalid.
   Refresh tokens are hashed (SHA-256) before storage.
   Token reuse detection: if the same refresh token is used twice,
   ALL tokens for that user are invalidated.

2. NOSQL INJECTION PREVENTION
   All request bodies/queries are sanitized by removing $ operator keys.
   Prevents attacks like { "email": { "$gt": "" } } which would match all.

3. RATE LIMITING (5 layers)
   - Global: 1000 req/15min per IP
   - Auth: 10 req/15min per IP
   - OTP send: 3 req/15min per email
   - OTP verify: 5 req/15min per email
   - Like: 100 req/hour per user
   - Message: 200 req/hour per user

4. IMAGE VALIDATION (2 layers)
   - MIME type check (Multer file filter)
   - Magic bytes check (actual file content analysis)
   Prevents uploading malicious files disguised as images.

5. INPUT VALIDATION (Joi schemas)
   Every request body is validated against a strict schema.
   Unknown fields are stripped. Invalid data returns 400.

6. CORS LOCKDOWN
   Production: only specified origins can make requests.
   Development: all origins allowed for convenience.

7. SECURITY HEADERS (Helmet)
   X-Content-Type-Options, X-Frame-Options, HSTS,
   Content-Security-Policy, and 10+ other headers.

8. FILE SIZE LIMITS
   Request body: 10MB max
   Individual photo: 5MB max
   Socket payload: 64KB max

9. AUTHORIZATION CHECKS
   - Gender-based route access (girls can't see queue, boys can't swipe)
   - Ownership checks (can't accept someone else's like)
   - Match participation checks (can't read messages in someone else's match)

10. SENSITIVE DATA HANDLING
    - Passwords: not used (OTP or Google auth)
    - Refresh tokens: SHA-256 hashed in DB
    - Logs: authorization headers and tokens redacted
    - Sentry: auth headers and cookies scrubbed
    - Error responses: no stack traces in production

11. ACCOUNT DELETION
    Fully GDPR-compliant deletion: removes ALL user data including
    photos, messages, matches, likes, blocks, and the user document.

12. BLOCK SYSTEM
    Blocks remove all connections (matches, likes) and prevent
    future visibility in both directions.
